# PR052: Рефакторинг leaderSendAEs и исправление deadlock в Stop

## Обзор изменений

PR52 содержит два независимых изменения в файле `pkg/raft/raft.go`:

1. **Рефакторинг `leaderSendAEs`** — устранение захвата переменной цикла в замыкании
2. **Исправление `commitChanSender`** — устранение deadlock при остановке (Bug-051)

Оба изменения прошли все 35 юнит-тестов и линтер без ошибок.

---

## 1. Рефакторинг `leaderSendAEs` (строки 648–255)

### Проблема

Исходный код использовал горутину внутри цикла по `peerIds`:

```go
for _, peerID := range cm.peerIds {
    go func() {
        // использует peerID — захватывает переменную цикла
    }()
}
```

В Go переменная цикла `peerID` переиспользуется на каждой итерации. Замыкание захватывает **ссылку** на неё, а не значение. К моменту выполнения горутины `peerID` может уже указывать на следующий элемент. Это классическая ошибка захвата переменной цикла (loop variable capture).

### Исправление

Рефакторинг разбил одну большую вложенную функцию на три отдельных метода:

#### `nextIndexArgsEntries` (новая, строка 12)

Вычисляет `nextIndex`, `AppendEntriesArgs` и список записей для отправки одному соседу. Работает под блокировкой `cm.mu`.

```go
func (cm *ConsensusModule) nextIndexArgsEntries(peerID, savedCurrentTerm int) (int, AppendEntriesArgs, []LogEntry) {
    cm.mu.Lock()
    defer cm.mu.Unlock()
    ni := cm.nextIndex[peerID]
    prevLogIndex := ni - 1
    prevLogTerm := -1
    if 0 <= prevLogIndex && prevLogIndex < len(cm.log) {
        prevLogTerm = cm.log[prevLogIndex].Term
    }
    var entries []LogEntry
    if ni < len(cm.log) {
        entries = append([]LogEntry{}, cm.log[ni:]...)
    } else {
        entries = nil
    }
    return ni, AppendEntriesArgs{
        Term:         savedCurrentTerm,
        LeaderID:     cm.id,
        PrevLogIndex: prevLogIndex,
        PrevLogTerm:  prevLogTerm,
        Entries:      entries,
        LeaderCommit: cm.commitIndex,
    }, entries
}
```

#### `leaderSendAEsToPeer` (новая, строка 71)

Отправляет AppendEntries одному соседу и обрабатывает ответ. Принимает `peerID` и `savedCurrentTerm` как параметры — **никакого захвата переменных цикла**.

Логика полностью идентична старому коду, но вынесена в отдельную функцию с явными параметрами.

#### `leaderSendAEs` (изменённая, строка 243)

Старая версия захватывала блокировку, проверяла состояние и запускала горутины. Новая версия делает то же самое, но запускает `leaderSendAEsToPeer` вместо анонимной функции:

```go
func (cm *ConsensusModule) leaderSendAEs() {
    cm.mu.Lock()
    if cm.state != Leader {
        cm.mu.Unlock()
        return
    }
    savedCurrentTerm := cm.currentTerm
    cm.mu.Unlock()

    for _, peerID := range cm.peerIds {
        go cm.leaderSendAEsToPeer(peerID, savedCurrentTerm)
    }
}
```

### Дополнительные улучшения

- **Устранение дублирования**: вычисление `nextIndex`, `prevLogIndex`, `prevLogTerm` и `entries` было продублировано в каждой горутине. Теперь это одна функция.
- **Читаемость**: логика отправки одному соседу изолирована, не перемешана с циклом и запуском горутин.
- **Безопасность**: явная передача `peerID` и `savedCurrentTerm` как параметров исключает ошибку захвата.

## 2. Исправление `commitChanSender` (строки 257–294)

### Проблема (Bug-051)

`Stop()` закрывает `newCommitReadyChan` и ждёт завершения `commitChanSender` через `WaitGroup`.
Если `commitChanSender` заблокирован на отправке в `commitChan` (канал заполнен),
он не может выйти из цикла, `wg.Done()` не вызывается — deadlock.

### Исправление

Два изменения в `commitChanSender`:

#### 2.1 Цикл приёма сигналов

**Было:** `for range cm.newCommitReadyChan`
**Стало:**
```go
for {
    _, ok := <-cm.newCommitReadyChan
    if !ok {
        return
    }
    // ...
}
```

Явная проверка закрытия канала позволяет немедленно выйти, если `Stop()` закрыл `newCommitReadyChan`.

#### 2.2 Отправка записей

**Было:** `cm.commitChan <- CommitEntry{...}` (блокирующая)
**Стало:**
```go
select {
case cm.commitChan <- ce:
case <-cm.newCommitReadyChan:
    return
}
```

Если `commitChan` заполнен и `Stop()` закрывает `newCommitReadyChan`, второй case активируется, горутина выходит, deadlock устранён.
